# UI Factory - Product Vision

## Overview

**UI Factory** is a browsable catalog of polished, single-page UI designs.

Each catalog entry presents one complete page as a concrete visual specimen. A user can inspect the design, evaluate its responsive and interaction behavior, and copy or download a curated `DESIGN.md` that translates the page's visual language into rules for an entire application.

The primary users are:

- The project owner, who browses and chooses a visual direction for a new application
- AI coding agents, which receive the exported `DESIGN.md` and use it as the shared design reference during implementation

UI Factory should reduce design churn. Instead of asking agents to invent and compare many directions, the user selects an existing design and gives every agent the same explicit guidance.

## Product Promise

> Find a page design that fits a new application, then export a self-contained `DESIGN.md` that AI coding agents can follow across the entire application.

UI Factory is not primarily an AI design generator, component library, or application builder. It is a visual catalog and design handoff tool.

## Core Workflow

1. Open UI Factory without creating an account
2. Browse or filter the available page designs
3. Open a design and inspect its full-page preview
4. Preview relevant responsive layouts and interaction states
5. Review the design rationale and intended use
6. Copy or download its `DESIGN.md`
7. Add that file to a new application repository
8. Instruct AI coding agents to follow it when building every page

The user chooses the design. Automated agent selection may be added later, but it is not required for the initial product.

## Catalog Unit

Each catalog entry represents one **single-page design**, such as:

- Analytics dashboard
- Product detail page
- Authentication page
- Account settings page
- Checkout page
- Data-management page
- Marketing landing page
- Mobile application shell

The selected page is both:

1. A concrete implementation that can be previewed and inspected
2. A visual specimen whose rules can be extended to pages not shown in the original design

A design does not need to contain an entire application or a complete component library.

## What Each Design Includes

Every published design should include:

- Stable design ID and version
- Title and concise description
- Page type and intended application types
- Visual style, theme, density, and platform metadata
- Full-page working preview
- Desktop, tablet, and mobile behavior where applicable
- Relevant loading, empty, error, validation, disabled, and success states
- Design rationale and usage guidance
- Source files, fixtures, tokens, and assets used by the preview
- Accessibility and interaction notes
- Curated, self-contained `DESIGN.md`
- Quality status such as `draft`, `reviewed`, `production-ready`, or `deprecated`

Only `production-ready` designs should appear in the default public catalog or offer an unqualified `DESIGN.md` export. Draft and reviewed entries remain in the authoring workflow. Deprecated entries may retain stable URLs for reference but should be excluded from normal discovery and clearly identify a replacement when one exists.

Metadata should use a small controlled vocabulary for core filters and optional free-form tags for additional discovery.

## The `DESIGN.md` Handoff

`DESIGN.md` is the primary export and the most important product artifact.

It should be authored and reviewed alongside the page design, not generated by AI when the user clicks the export button. The catalog should copy or download the validated file exactly as published.

Each exported `DESIGN.md` should contain:

1. Design name, version, and visual intent
2. Canonical page reference
3. Design principles
4. Color system and semantic color roles
5. Typography rules and scale
6. Spacing, sizing, density, borders, radii, and elevation
7. Layout and composition rules
8. Navigation patterns
9. Component appearance and behavior
10. Responsive behavior
11. Interaction and motion guidance
12. Accessibility requirements
13. Rules for extending the design to new pages
14. Explicit do and do-not guidance
15. An acceptance checklist for AI coding agents

The extension rules are essential. They should explain how an agent can create new settings, authentication, forms, tables, detail pages, and other screens while preserving the selected specimen's visual language.

The export must be understandable without access to UI Factory's source code. Consumers should not need UI Factory or the Impeccable skill installed to use it.

## Discovery Experience

UI Factory should optimize for fast visual selection rather than exhaustive metadata browsing.

Users should be able to:

- Scan large, representative screenshots
- Browse by page type
- Filter by visual style, theme, density, and platform
- Search titles, descriptions, and tags
- Open a stable, shareable design URL
- Preview responsive layouts
- Inspect relevant page states
- Copy or download `DESIGN.md` with one clear action
- View related alternatives

The catalog should include multiple visual directions for important page types. A collection containing several meaningfully different dashboards is more useful for choosing a direction than a collection containing only one dashboard and unrelated page examples.

Advanced search, recommendation ranking, and side-by-side comparison should be added only when catalog size and usage demonstrate a need.

## Design Authoring with Impeccable

UI Factory and every catalog design should use the **Impeccable** skill during design creation and review.

The authoring workflow should be:

1. Define the page's product context, intended audience, and visual intent
2. Use Impeccable to shape the design before implementation
3. Build the representative page
4. Review responsive behavior, accessibility, interaction states, and visual consistency
5. Run an Impeccable critique and technical audit
6. Polish the design before marking it production-ready
7. Curate the final agent-facing `DESIGN.md`

Impeccable should be used for:

- The UI Factory browsing experience itself
- Every individual page design added to the catalog
- Design critique, accessibility review, responsive review, and final polish

Each catalog design should keep its design context independent so that one entry's visual language does not unintentionally influence another.

## Quality Standard

A smaller catalog of complete, trustworthy designs is more valuable than a large catalog of shallow examples.

A design should not be marked production-ready until it has:

- A coherent and intentional visual direction
- A working preview using realistic fixture content
- Structural responsive behavior rather than simple scaling
- Keyboard-accessible interactions and visible focus states
- A WCAG 2.2 AA accessibility baseline
- Reduced-motion behavior where motion is used
- Relevant loading, empty, error, and validation states
- Declared dependencies, assets, and licenses
- A self-contained `DESIGN.md`
- At least one automated preview or interaction smoke test
- Completed Impeccable critique, audit, and polish passes

Every design should also state when it should be used, when it should be avoided, and what trade-offs it makes.

## Guiding Principles

### 1. Human selection first

The user should remain in control of choosing the visual direction. AI agents follow the exported specification rather than inventing alternatives by default.

### 2. Single-page specimens

Each entry should communicate one visual direction through a complete page. Supporting documentation may generalize that direction, but the catalog unit remains simple and understandable.

### 3. Curated exports

`DESIGN.md` files should be deliberate design specifications, not variable AI-generated summaries.

### 4. Keep designs independent

Each design may define its own components, tokens, styles, assets, fixtures, interactions, and dependencies. Shared infrastructure must not force every entry into the same visual system.

### 5. Optimize for visual discovery

Screenshots and live previews should lead. Metadata should support discovery without overwhelming the browsing experience.

### 6. Quality over quantity

New designs should add meaningful visual choices, not minor variations created only to increase catalog size.

### 7. Build iteratively

The project should begin with the smallest complete browse, inspect, and export workflow. Expansion should follow observed needs.

### 8. Prefer native capabilities

Use native SvelteKit, browser, and platform capabilities before adding dependencies or custom infrastructure.

## Initial Technology Direction

The first version will use:

- SvelteKit
- Svelte 5
- TypeScript
- Tailwind CSS
- Vite
- Zod
- Vitest
- Playwright

The preview implementations are Svelte-first. The exported design guidance should remain useful regardless of the framework used by the destination application.

## Initial Product Scope

The first production-quality version should support:

1. Browsing a visual gallery of designs
2. Searching and filtering by a small set of useful facets
3. Opening a stable design detail page
4. Previewing desktop, tablet, and mobile behavior
5. Inspecting relevant page states
6. Reading design rationale and structured metadata
7. Copying `DESIGN.md`
8. Downloading `DESIGN.md`
9. Adding designs through a predictable folder structure
10. Automatically discovering and validating design entries
11. Running quality checks before publication

Catalog data should come from the same validated design entries used by the application. A separate agent API is not required for the first release.

## Initial Sample Designs

The first release should prioritize meaningful visual comparison. Suggested examples are:

1. Compact neutral SaaS analytics dashboard
2. Editorial analytics dashboard
3. Dark operational dashboard
4. Soft consumer dashboard
5. High-conversion e-commerce product detail page
6. Minimal authentication page

Each example should be complete enough to prove the preview, responsive, state, metadata, quality, and `DESIGN.md` workflow.

## Success Criteria

UI Factory is successful when:

- A user can find a suitable visual direction in under ten minutes
- A user can copy or download `DESIGN.md` without setup or an account
- The exported file can be used in a new repository without additional explanation
- Separate AI coding agents produce visually consistent pages from the same file
- New pages remain recognizably faithful to the selected specimen
- A new design can be added without changing the catalog's core application code
- Published designs consistently meet the defined quality standard

Total design count is not a primary success metric.

## Non-Goals for the First Release

The first release should not include:

- AI-generated designs
- AI-generated `DESIGN.md` exports
- Autonomous design selection
- Vector databases or embeddings
- A visual page builder
- Full application generation
- Multi-framework source-code generation
- User accounts or collaboration features
- Complex plugin infrastructure
- Large-scale token transformation pipelines

These capabilities should be considered only when the core browse, choose, and export workflow is proven.

## Long-Term Outcome

UI Factory should become an expanding, versioned catalog where a user can browse all available page designs, choose one that fits a new application, and export reliable instructions for AI coding agents.

The catalog may eventually support direct agent access and recommendations, but human visual selection and a trustworthy `DESIGN.md` handoff should remain the foundation.

UI Factory should not attempt to represent every possible design. It should provide a growing set of distinct, polished, reusable visual directions that reduce design uncertainty and help agents build coherent applications.
